# Совместная работа

Этот документ нужен, чтобы новый участник быстро понял, как безопасно вносить изменения в проект.

## Перед началом

1. Установить зависимости: `npm ci`.
2. Создать `.env.local` из `.env.example`.
3. Сгенерировать Prisma Client: `npm run db:generate`.
4. Запустить проект: `npm run dev`.

## Рабочий цикл

1. Создавайте отдельную ветку под задачу.
2. Перед правками проверьте актуальные файлы и не перетирайте чужие изменения.
3. Для UI-правок проверяйте desktop и mobile.
4. Для auth, ролей и данных проверяйте серверную часть, не только клиентский экран.
5. Перед PR запускайте `npm run typecheck` и `npm run build`.

## Проверки

```bash
npm run typecheck
npm run build
npm run lint
```

`npm run verify` запускает typecheck и production build. Линт вынесен отдельно, потому что в проекте есть известные lint-долги, описанные в `docs/known-issues.md`.

## Работа с базой

Локальная demo DB лежит в `dev.db`.

```bash
npm run db:migrate
npm run db:seed
```

Не коммитьте изменение `dev.db`, если задача не была явно про demo-данные.

## Code style

- TypeScript strict mode включён.
- Предпочитать server components и server actions там, где не нужен браузерный state.
- `use client` держать как можно ниже по дереву.
- Не добавлять новую бизнес-логику только в mock-data, если она должна жить в БД.
- Для новых защищённых действий проверять текущего пользователя на сервере.

## Что не коммитим

- `.env.local` и любые секреты.
- Сгенерированные скриншоты.
- HyperFrames-рендеры и презентационные артефакты.
- Локальные agent/cache директории.

## PR-чеклист

- Описано, что меняется и почему.
- Указаны затронутые роли: работодатель, соискатель, администратор.
- Приложены скриншоты для заметных UI-изменений.
- Выполнены `npm run typecheck` и `npm run build`.
- Если `npm run lint` не проходит, указана причина и ссылка на известную проблему.

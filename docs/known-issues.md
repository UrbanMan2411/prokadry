# Известные проблемы

Этот список помогает команде не терять уже найденные риски и планировать стабилизацию.

## P1: работодатель получает доступ до модерации

`app/actions/auth.ts`

Работодатель создаётся со статусом `PENDING`, но сразу получает сессию `EMPLOYER` и попадает в dashboard. Нужно решить продуктово:

- показывать pending-экран до проверки;
- или разрешать только ограниченный профиль;
- или блокировать ключевые действия до `Employer.status === APPROVED`.

## P1: сессия не сверяется с актуальным пользователем

`proxy.ts`, `app/dashboard/page.tsx`

JWT-cookie доверяется до истечения срока. Если пользователя отключили или поменяли роль, старая cookie всё ещё может работать. Для защищённых server routes нужно сверять `userId` с БД и проверять `isActive`, роль и статус работодателя.

## P1: production может уйти в demo SQLite

`lib/db.ts`, `prisma.config.ts`

Сейчас есть fallback на `file:./dev.db`, а `lib/db.ts` умеет копировать DB в `/tmp`. Для production нужно требовать явный `DATABASE_URL` и не использовать временную SQLite-базу.

## P2: lint пока не зелёный

Текущие типы проблем:

- компоненты объявлены внутри render в `components/employer.tsx` и `components/regions.tsx`;
- sync `setState` внутри `useEffect` в нескольких client-компонентах;
- отдельные warnings по неиспользуемым переменным и шрифтам.

До включения `lint` в обязательный CI нужно почистить эти места.

## P2: нет защиты от перебора паролей

`app/actions/auth.ts`

Форма входа должна получить rate limit по IP/email, audit trail неуспешных попыток и, при необходимости, мягкую задержку или временную блокировку.

## Рекомендуемый порядок стабилизации

1. Зафиксировать модель доступа для `PENDING` работодателей.
2. Добавить серверную проверку активного пользователя и статуса роли.
3. Убрать production fallback на demo DB.
4. Почистить lint-долги.
5. Добавить базовые тесты auth/role flows.
